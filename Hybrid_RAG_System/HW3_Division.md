### 🧑‍💻 同学 A：数据工程师 (Data Engineer)
**核心目标：** 搞定“知识库”，为 RAG 提供高质量的检索来源。
* 
**任务 1：网页爬虫与清洗。** 编写脚本爬取 `eecs.berkeley.edu` 域名下的网页（忽略 PDF 和图片等非文本文件）。可以使用提供的建议工具如 `Beautifulsoup4` 或 `resiliparse` 。
* 
**任务 2：文本切片 (Chunking)。** 将清洗后的长 HTML 文本切分成适合检索和输入给大模型的片段（Passages/Chunks）。
* 
**任务 3：参与交叉标注 (前期配合同学 C)。** 抽出几个小时，配合同学 C 标注测试集中 30% 的数据，用于计算 IAA（标注者间一致性）。
* 
**后期任务 (3/17 之后)：** 官方会在 3/17 晚上 7 点发布参考语料库 (reference retrieval corpus) 。你需要负责对比咱们自己爬的语料库和官方语料库，看看哪个效果好，并整理出差异（这直接对应报告的 Q2）。
### 🛠️ 同学 B：RAG 架构师 (RAG Architect)

**核心目标：** 搭建系统管道，确保代码符合 Gradescope 严苛的性能限制。
* 
**任务 1：搭建本地环境与检索索引。** 严格按照文档中的库版本（Python 3.10.12, torch>=2.0.0, faiss-cpu>=1.7.4 等）配置环境 。搭建本地的 BM25 (稀疏检索) 或 FAISS (稠密检索) 索引 。**注意：** 如果用嵌入模型 (Embedding model)，参数必须 $\le$ 400M，模型文件大小 $\le$ 400MB 。
* 
**任务 2：实现 `run.sh` 管道。** 编写符合 Autograder 要求的 `run.sh` 脚本，必须能接收两个参数（输入问题文件路径和输出预测文件路径）。
* 
**任务 3：LLM 接入与性能优化。** 严格使用官方提供的 `llm.py` 调用大模型（如 `meta-llama/llama-3.1-8b-instruct`）。增加超时处理 (timeout exception)，防止 API 卡死 。
* 
**硬性底线：** 必须确保代码能在只有 4GB 内存、无 GPU 的环境下运行，且每个问题平均响应时间控制在 0.6 秒以内（本地测试最好能在 2 分钟内跑完 100 题）。
### 📊 同学 C：评测专家兼项目统筹 (QA & PM)

**核心目标：** 掌控数据集质量，评估模型表现，并主笔最终报告。
* 
**任务 1：构建 QA 数据集 (最紧急！)。** 手动浏览 EECS 网站，撰写至少 100 个问答对 。要求：答案必须在 10 个词以内 ，必须是纯文本/表格能找到的 。可以包含最多 5% 的 Yes/No 问题和 5% 的简单推理问题 。
* 
**任务 2：Prompt 工程。** 配合同学 B，设计给 LLM 的提示词 (Prompt)，让它能根据检索到的文本生成极简的短答案（避免它废话连篇导致指标 F1 降低）。
* 
**任务 3：模型评测与消融实验 (Ablations)。** 运行评测脚本，计算 Exact Match 和 F1，同时建议计算“召回率 (Recall)”来判断是检索出了问题还是生成出了问题 。配合同学 B 跑出两组消融实验（比如对比不同切片大小、或 BM25 对比 FAISS）。
* 
**任务 4：错误分析与撰写报告。** 找出 F1=0.0 的失败案例进行分类分析 。统筹撰写最终的 2 页英文报告 。
